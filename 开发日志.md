### RabbitMQ / Worker 升级

- RabbitMQ 消息保持 v0 设计，只投递：
  - `{"run_id": "run_xxx"}`
- 明确 RabbitMQ 只负责通知，PostgreSQL 才是事实来源，不把 `user_id`、`workspace_id`、`message` 塞进消息体。
- Worker 消费 run_id 后新增安全判断：
  - 消息体非法：记录日志并 ack。
  - run 不存在：ack。
  - run 状态不是 `queued`：ack，防止 RabbitMQ 重复投递导致重复执行。
  - run 是 `queued`：写 `worker.run.received`。
  - 将 run 状态改为 `running`，并写 `run.started`。
  - 调用 `AgentOrchestrator.run(run_id)`。
  - Worker 自身异常时将 run 标记为 `failed`，写 `run.failed`，然后 ack。
- `AgentOrchestrator` 不再负责把 run 从 `queued` 改成 `running`，避免和 Worker 重复写 `run.started`。

### v1.5 Worker 启动重试与 Streaming 卡住排查

  - `rotom-worker` 曾在 RabbitMQ 端口短暂不可连时启动失败并退出。
  - 队列里有消息但没有消费者执行 run，前端 SSE 就会一直等待新 chunk。
- `worker.py` 增加 RabbitMQ 连接重试循环：
  - 首次连接失败不再直接退出。
  - 每 5 秒重试一次，直到连接成功。
- `deploy/docker-compose.yml` 为 worker 增加 `restart: unless-stopped`，避免 Worker 异常退出后无人消费队列。
- RabbitMQ 当前开发端口：
  - AMQP：`5672`
  - 管理页面：`15672`
- 验证：
  - `python3 -m compileall -q backend` 通过。
  - `docker compose -f deploy/docker-compose.yml up -d --build worker` 通过。
  - RabbitMQ 队列 `agent_runs` 当前 `consumers=1`、`messages=0`。
